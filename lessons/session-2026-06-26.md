# Lessons Learned — Session 2026-06-26

## 1. Smart measure scrolling during playback

The original `_updatePlayHighlight` scrolled using `block: 'nearest'`, which only scrolled when the measure was entirely out of view. Musicians need to see ahead during playback, but excessive scrolling interferes with visual tracking.

Fix: `_scrollToMeasure(idx)` checks if the playing measure is within 2 measure-heights of the container bottom. If so, scrolls it to the top (`block: 'start'`); otherwise uses minimal scroll (`block: 'nearest'`). Also added scrolling to `_updateRecordingHighlight` which had none.

## 2. GitHub Pages deployment: standard Go WASM, not tinygo

`tinygo build -target wasm` emits a WASI binary with `wasi_snapshot_preview1` imports that browsers don't provide. The error was "module is not an object or function" at `WebAssembly.instantiate()`.

Fix: Use `GOOS=js GOARCH=wasm go build` which targets the browser's native WASM host with the `wasm_exec.js` glue. The binary is larger (4.7MB vs 956KB) but works everywhere.

Added `make wasm` and `make gh-pages` targets. The deploy script copies web files to a temp directory before switching to the `gh-pages` orphan branch, avoiding the "no such file or directory" error when the source branch's `web/` directory disappears.

## 3. gh-pages deploy script: copy before branch switch

The script copied `web/*` files after switching to `gh-pages`, but those files only exist on the source branch. After the orphan branch checkout, `web/` was gone.

Fix: Copy web files to a `mktemp -d` staging directory before switching branches.

## 4. Safari AudioContext must be explicitly resumed

Safari (macOS and iOS) creates AudioContext in "suspended" state by default. Chrome auto-resumes on first use. Without explicit `resume()`, the scheduled WebAudio notes never play, and Safari's `<audio>` element refuses to play recorded blobs.

Fix: Call `audioCtx.resume()` in two places:
- Immediately after creation in `initSoftSynth()` — within the user-gesture chain
- Again in `play()` before scheduling — Safari can re-suspend after async work

## 5. Safari doesn't support audio/webm playback

`MediaRecorder` on Safari produces `audio/mp4`, not `audio/webm`. The recording blob was hardcoded as `'audio/webm'`, which Safari's `<audio>` element can't decode. This caused "Playback failed" on macOS Safari.

Fix: Detect supported MIME types with `MediaRecorder.isTypeSupported()`, trying `audio/webm;codecs=opus` → `audio/webm` → `audio/mp4`. Store the detected type in `_recMimeType` and use it for both the `MediaRecorder` constructor and the recording blob.
