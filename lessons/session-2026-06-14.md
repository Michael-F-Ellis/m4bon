# Lessons Learned — Session 2026-06-14

## 1. cmd() and runCommand() are different APIs

MuseScore 4's plugin API has two command dispatchers:

| Call | Signature | Works for |
|------|-----------|-----------|
| `cmd("action-name")` | Global function | The proven approach. Dispatches by action's internal name from shortcuts/preferences. |
| `curScore.runCommand("action")` | Score method | Unreliable for our use case. May dispatch differently or need different command names. |

Switching from `curScore.runCommand("tie")` to `cmd("tie")` was necessary — but even `cmd("tie")` failed for our note-insertion workflow.

## 2. QML "read-only" properties silently ignore writes

The docs say `Note.tieFor` is read-only. When you write:

```javascript
note.tieFor = otherNote;   // no error, no effect
```

It throws no exception but does nothing. Same for many C++ → QML property bindings. Always test with a trivial plugin before building complex logic around property setters.

## 3. newElement(Element.TIE) can crash MuseScore

Community reports confirm that `newElement(Element.TIE)` + `note.add(tie)` crashes MuseScore 4. The `Tie` element requires `startNote` and `endNote` links that aren't properly exposed in the QML layer. Avoid creating Tie elements programmatically.

## 4. Tuplet API: addTuplet() required, not setDuration() alone

Calling `setDuration(1, 12)` (triplet eighth) or `setDuration(1, 6)` (quarter-note triplet) produces wrong results because `setDuration()` takes the **nominal (visual)** note value, not the actual tick duration.

Correct approach:
```javascript
cursor.addTuplet(fraction(3, 2), fraction(1, 4));  // create tuplet container
cursor.setDuration(1, 8);                            // nominal eighth
cursor.addNote(pitch, false);                        // × 3
```

The tuplet detection logic (`isStandardDuration()`) works by checking whether the per-note fraction reduces to a denominator that's a power of 2 with numerator 1 or 3.

## 5. Standalone sustain groups were silently dropped

When `-` is its own group (e.g. `a - b -`), `countActivePositions()` returns 0 and the group was skipped entirely in `resolveDurations`. Fix: detect sustain-only groups (all slots are `"sustain"`) and extend the last event by the full group duration before continuing.

## 6. Pipeline architecture pays off

The multi-pass approach made every change surgical:

```
normalize → tokenize → parseGroup → resolveDurations → splitDurations → resolveOctaves → insertNotes
```

Adding tuplet detection changed only `resolveDurations`. Fixing sustain-only groups changed only `resolveDurations`. Adding a field to events propagates naturally. Contrast with a monolith where every tweak risks breaking unrelated logic.

## 7. Symbol constants make the DSL forkable

Putting all DSL characters as `property string` at the top of `MuseScore`:

```javascript
property string symSharp : "#"
property string symRest  : ";"
```

Means someone can clone the plugin, change the constants, and use completely different symbols without touching parser logic. This came from the user's explicit requirement.

## 8. Non-standard durations need splitting for MuseScore

Even when durations are mathematically correct (5/8 = 2.5 beats), `setDuration(5, 8)` may render incorrectly or crash because 5/8 isn't a standard note value. The splitter converts to standard values (1/2 + 1/8) via greedy subtraction.

But the greedy approach doesn't respect engraving rules (barline visibility, beat boundaries). A proper version must track the cursor's beat position within the measure — deferred to future work.

## 9. The "\_split" field is a clean continuation marker

Rather than coupling the splitter output to the insert logic, `splitNonStandardDurations` marks continuations with `_split: true`. Downstream code can check `ev._split` to behave differently without knowing about the splitter's internals.

## 10. Deferral is better than broken automation

After 8 failed approaches to create ties programmatically, the user made the right call: produce correct durations and positions, let the user tie manually in the MuseScore UI (select note + click T). Broken tie attempts produced worse output (missing notes, empty windows) than no tie attempt at all.

## 11. MuseScore 4.7 plugin development is heavily constrained

Key limitations discovered when trying to build a CLI-based test cycle:

| Limitation | Impact |
|---|---|
| `--plugin` is a Qt QPA flag, not for QML plugins | Can't load plugins from CLI |
| `-j job.json` crashes with Chromium mutex error in headless mode | Can't use converter mode for plugin testing |
| `--qmljsdebugger` exists but requires Qt Creator | Can't do remote debugging easily |
| `XMLHttpRequest` with `file://` blocked in QML sandbox | Can't read local files from plugin |
| Non-dialog plugins (`onRun` without `pluginType`) may not fire in MS4.7 | Script-type plugins unreliable |
| "Reload plugins" button doesn't actually re-read changed .qml files | Must restart MuseScore for every change |
| Dialog `onRun` fires but `cursor.rewind(1)` needs a pre-existing selection | Plugin requires manual measure click before use |
| `plugins.json` stores shortcuts/prefs but not enabled/disabled state | Enabled state stored in `extensions/config.json` instead |
| Process name in AppleScript is `mscore` not `MuseScore 4` | AppleScript targeting requires the right process name |
| New plugins show in Plugin Manager but are DISABLED by default | Must manually enable before they appear in Plugins menu |

## 12. MusicXML is a far better output target than the MuseScore QML API

Switching from QML plugin to Go CLI + MusicXML eliminated every pain point:

| Before (QML plugin) | After (Go CLI + MusicXML) |
|---|---|
| 8-25s edit-test cycle with MuseScore restart | 0.4s with `go test`, instant with `./m4bon` |
| Ties require MuseScore API hacks | MusicXML has native `<tie>` elements |
| `console.log()` only visible in terminal | stdout, pipe to diff, grep, jq |
| Can't inspect inserted notes programmatically | `xmllint`, `diff`, or validate against expected output |
| No test infrastructure | `go test` with table-driven tests |
| QML sandbox blocks file I/O | Full filesystem access |
| No version control of output format | MusicXML = plain text, diff-able |
| Single MuseScore API dependency | Zero runtime dependencies (Go static binary) |

## 13. Go's encoding/xml is well-suited for MusicXML generation

- Struct tags map directly to MusicXML element nesting
- `xml.MarshalIndent` produces clean, formatted output
- No external dependencies needed for the core feature set
- 480 DPPQ (divisions per quarter note) is the standard MIDI convention and cleanly divides all durations we produce (quarters, eighths, triplet eighths, etc.)
- MIDI-to-MusicXML note conversion: `octave = midi/12 - 1` (MIDI 60 = C4)

## 14. The parser port from JS to Go was straightforward

The QML/JS parser was written as pure functions with no MuseScore dependencies, which made the port to Go a direct 1:1 translation. Key types:

```
Token → parseGroup → ParseResult (slots)
                     → resolveDurations → []Event
                                        → splitNonStandardDurations → []Event
                                                                     → resolveOctaves → []Event (with MIDI)
```

The MusicXML generator then takes `[]Event` and produces `<score-partwise>` XML.

## 15. Duration fractions don't need reduction in the event pipeline

In `resolveDurations`, position durations like 3/24 (for a beat of 3/8 divided into 3 positions) are functionally equivalent to 1/8. The MusicXML generator works with ticks (480 * 4 * num / den) which produces the correct integer regardless of whether the fraction is reduced. For tests, use `gcd()` to compare reduced forms when checking expected values.

## 16. Go's xml.MarshalIndent produces `<tag></tag>` not `<tag/>`

When testing for XML elements in Go string output, match against `<tie type="start">` not `<tie type="start"/>`. The self-closing tag syntax is only used for elements with no content and no child elements in Go's XML encoder — but tie elements with attributes are rendered as full open/close tags.

## 17. TDD Infrastructure: DSL key/meter directives replace CLI flags

Removed the `-time` flag from the CLI. Time signatures (`M4/4`, `M6/8`) and key signatures (`KC`, `KE&`) are now parsed from the DSL itself via `stripDirectives()` in `pipeline.go`. Key sigs use circle-of-fifths encoding (`fifths` in MusicXML). This makes test cases self-documenting — each `.dsl` file starts with `M4/4` or `M6/8` instead of relying on external flags.

```go
// Before
ParseDSL(dsl, timeNum, timeDen int) DSLResult

// After
ParseDSL(dsl string) DSLResult  // K/M parsed from text
```

## 18. Verovio renders `<tie>` as sound but needs `<tied>` for visual ties

MusicXML has two tie-related elements:
- `<tie>` — indicates a tie for playback/sound purposes
- `<tied>` (inside `<notations>`) — the visual arc marking

Verovio (up to 6.2.1) does NOT render `<tie>` alone. The tie arc only appears when `<tied>` is present inside `<notations>`. Fix: emit both side by side.

```xml
<tie type="start"/>
<notations>
  <tied type="start"/>
</notations>
```

## 19. MusicXML element order matters for schema compliance

Go's `xml.MarshalIndent` emits struct fields in declaration order. The XSD requires `<voice>` before `<staff>` inside `<note>`.

## 20. Golden file tests make refactoring safe

The `golden_test.go` runner compares `.dsl → m4bon → MusicXML` output against `.expected.mxml` files. Key features:
- `-update-golden` flag overwrites expected files when intent changes
- Each case runs as a sub-test for clear failure attribution
- Schema validation (via `xmllint`) catches structural issues
- 5 initial cases cover: basic notes, sustain chains, tuplets, chords, compound meter

The `render.sh` script completes the cycle: DSL → MusicXML → SVG (Verovio) → PNG (rsvg-convert) → open.

## 21. Octave marks `^`/`/` apply to the next pitch, not the prior one

In the original parser, `^` and `/` were accumulated in `oct` and consumed by `emitNote` on the preceding pitch. `c^c` gave C5 C5 instead of C4 C5. Fix: track `pendingOct` separately — shifts after a letter go into `pendingOct`, transferred to `oct` when the next pitch letter triggers `emitNote`. Same pattern applied to accidentals with `pendingAcc`.

## 22. Beams and tuplet brackets need explicit MusicXML elements

Verovio does not auto-beam across beat boundaries. Beams must be emitted per beat group using `<beam number="1">begin|continue|end</beam>`, computed by tracking cumulative tick position. Tuplet brackets need `<notations><tuplet type="start|stop" number="1"/></notations>` on the first and last notes of each tuplet — the `<time-modification>` element alone is insufficient.

## 23. MusicXML `<accidental>` element is required for display

`<alter>` in `<pitch>` controls the sounding pitch but Verovio only displays an accidental sign when `<accidental>` is present as a sibling element. Without it, `#c` renders as plain C. Mapping: `1→sharp`, `-1→flat`, `2→double-sharp`, `-2→flat-flat` (Verovio 6.2.1 uses `flat-flat`, not `double-flat`).

## 24. Courtesy naturals require tracking pitch state

A `%d` after `#d` or `&d` needs `<accidental>natural</accidental>` because the same pitch class had a different accidental earlier. The generator tracks which pitch letters have had accidentals in a `pitchStates` map and emits courtesy naturals when returning to `alter=0`. This map must reset at each measure boundary when barline support is added.

## 25. Comment detection in SanitizeDSL must not eat sharps

`SanitizeDSL` was stripping any line starting with `#`, which ate `#c` (sharp C) when passed as the first argument. Fixed: only strip lines starting with `# ` (hash + space) or bare `#` — not `#c`, `#&b`, etc.

## 26. MusicXML `<dot/>` element is required for dotted durations

`noteTypeForDuration` correctly maps 3/8 to `"quarter"` but the note renders as a plain quarter without `<dot/>`. Go's `xml.MarshalIndent` renders `[]DotEl{ DotEl{} }` as `<dot></dot>` which Verovio and MuseScore interpret as an augmentation dot. The `dotCount()` helper derives dot count from the reduced fraction: numerator 3 → 1 dot, 7 → 2 dots, 15 → 3 dots.

## 27. Barline-aware duration splitting uses fractions, not ticks

The invisible-barline rule (no single note value may cross the midpoint of a 4/4 measure) required splitting notes at `timeNum/(2*timeDen)` of a whole note. Key design decision: use fraction arithmetic instead of tick-based position tracking.

| Approach | Risk |
|----------|------|
| Tick-based (DPPQ=480) | Fails for irregular tuplets where start/end ticks don't align to barline |
| Fraction-based | Works for any tuplet ratio, any time signature |

The `splitAtBarline` function runs *before* `splitNonStandardDurations` so raw (possibly non-standard) durations are split at the barline first, then each side is independently reduced to standard note values. This produces optimal notation — a 5/8 duration crossing the barline becomes 1/4 + 3/8 (dotted quarter) instead of 1/4 + 1/4 + 1/8.

## 28. Pipeline event ordering matters for split correctness

`ParseDSL` now runs:
```
resolveDurations → splitAtBarline → splitNonStandardDurations → resolveOctaves
```

The ordering is critical: `splitAtBarline` must see the unresolved durations (e.g. 5/8) so it can split them at the barline before `splitNonStandardDurations` greedily decomposes them into the largest standard chunks (1/2 + 1/8). Reversing the order produces extra tied fragments.

## 29. Verovio + imgcat creates a sub-second visual feedback loop

The `render.sh` pipeline — `m4bon → verovio → rsvg-convert → imgcat` — renders MusicXML to PNG and displays it inline in iTerm2 in under 300ms. This is faster than opening MuseScore and works in any terminal session, making it practical for both human and LLM-driven debugging. The shorthand notation (`Qa3 Qb3 - Q.b3 Ec4`) further accelerates communication by describing musical intent without parsing XML.

## 30. MusicXML `<part>` must contain all measures, not one per `<part>`

Verovio silently drops measures after the first `<part>` element. Each `<part>` must contain ALL its measures as children — not one `<part>` per measure. The fix: collect all measures into a slice, then emit one `<part id="P1">` with the full slice.

```go
// Before (broken — only first measure renders):
for ... {
    score.Parts = append(score.Parts, Part{ID: "P1", Measures: []Measure{meas}})
}

// After (correct):
var allMeasures []Measure
for ... {
    allMeasures = append(allMeasures, meas)
}
score.Parts = append(score.Parts, Part{ID: "P1", Measures: allMeasures})
```

## 31. Conditional `<attributes>` must omit unchanged clef to avoid courtesy clefs

When emitting `<attributes>` for a time-signature change (e.g. 4/4 → 3/4), including a `<clef>` element — even with the same value — causes Verovio to render a courtesy clef warning at the end of the previous measure. Fix: make `Attributes.Clef` a `*Clef` with `xml:",omitempty"` and only set it on measure 1 (or when clef actually changes).

```go
type Attributes struct {
    Clef *Clef `xml:"clef,omitempty"`  // was Clef Clef — always marshaled even zero-valued
}
```

## 32. `splitAtBarline` only applies to even-beat meters

The invisible-barline formula `timeNum / (2 * timeDen)` is designed for 4/4, 2/4, 6/8, etc. where the mathematical midpoint falls between two beats. For odd-beat meters (3/4, 9/8, 5/4), the midpoint lands inside a beat, producing incorrect splits (e.g. a quarter note in 3/4 split into two tied eighths). Fix: return early when `timeNum % 2 != 0`.

## 33. Cross-measure sustain needs `priorEvent` in two code paths

The `-` sustain operator at the start of a measure triggers two separate code paths in `resolveDurationsWithPrior`:

| Path | Condition | Fix |
|------|-----------|-----|
| Sustain-only group | `activeCount == 0` | Create new event copying `priorEvent` letter/pitches |
| Mixed group (`-f`) | `activeCount > 0`, first slot sustain | Same — create seed event from `priorEvent` |

Both paths need `priorEvent` from the previous measure's last note/chord, passed through new function `resolveDurationsWithPrior()`. Without the second path, `BQ. -f g` crashes with "sustain with no prior note across groups".

## 34. Cross-measure sustain marks `TieNext` on the predecessor

When a sustain creates a `Split=true` event in the next measure, the previous measure's last event needs a matching tie-start. The pipeline sets `ev.TieNext = true` on the predecessor after detecting `events[0].Split` in the new measure. The generator then emits `<tie type="start">` when `ev.TieNext` is true.

## 35. Chord pitch ordering must not use alphabetical sort

`(dfa)` is a valid D minor triad (D, F, A). The letters `a` and `f` are not in alphabetical order, but the pitches ascend. The original check `chordPitches[p].Letter <= chordPitches[p-1].Letter` rejected this. Fix: remove the alphabetical check entirely — chord voicing is handled by octave resolution.

## 36. Chord octave resolution uses "next higher" not "nearest"

Subsequent chord tones must be the next octave higher with that pitch name, not the nearest octave to the reference. `(dfa)` after C4 should resolve as D4→F4→A4. With the "nearest" approach, F would resolve to F4 (correct) but A would resolve to A4 (correct by luck). The proper approach uses `nextHigherPitch()` which finds the first octave strictly above the previous chord tone.

For split-continuation chords (across barlines or invisible barlines), Midis must be copied from the predecessor event to keep the same voicing. The copy logic checks both within-measure (`i > 0`) and cross-measure (`mi > 0`) predecessors.

## 37. Chord notes in MusicXML generator were missing Tie and Notations

The `EventChord` case in `Generate()` constructed `NoteEl` values without `Tie` or `Notations` fields. This silently dropped all tie and notation information for chord notes — split chords appeared without ties. Fix: add `Tie: ties, Notations: notations` to the chord `NoteEl` literal, matching the `EventNote` case.

## 38. Measure-level metadata requires per-measure state tracking

The generator now tracks `prevTimeNum`, `prevTimeDen`, and `prevFifths` across measures. Auto-detection of time signature changes (via `totalTicks` reduction) happens after `resolveDurations` when no explicit directive is given. The auto-detection uses `gcd(actualTicks, TicksPerWholeNote)` to derive the reduced time signature. This only triggers in multi-measure mode (when `|` is present) to preserve backward compatibility with single-measure fragments.

## 39. `notify.sh` is the standard completion signal for plan execution

After any plan implementation completes, `bash notify.sh "summary message"` sends a text notification. This enables async review — the user doesn't need to watch the terminal for completion. The notification should include pass/fail status and key metrics (test count, test duration).

## 40. `./tmp/` directory for user-placed artifact inspection

The user creates `./tmp/` for placing output files (`bug-case.mxml`, etc.) that the assistant can inspect. This directory is gitignored. The convention accelerates debugging — the user saves output, the assistant reads and analyzes the file directly instead of re-running commands.
