## Key Topics

- Measure Actions Popover + Action Bar: tap measure number in rendered display → popover with Edit/Add Comment/Clear Comments
- Persistent action bar between textarea and rendered display that mirrors popover actions for cursor-linked measure
- Went through a design exploration before settling: rejected ContentEditable display, rejected textarea overlay, landed on measure-number-as-trigger approach
- All changes were purely frontend except `data-idx` attribute threaded through Go HTML renderer
- Zero changes to parser, pipeline, or render logic — `!` comment system already worked end-to-end

## Design Decisions

1. **Measure number as trigger** — least visual clutter, measure number already distinguishes itself via color (`var(--muted)`), taps don't conflict with other measure content interactions

2. **Popover + Action Bar (dual entry points)** — popover for "I see the measure I want" (tap on it), action bar for "my cursor is on this measure" (always visible). Both call the same underlying methods.

3. **Rejected: ContentEditable display** — HTML soup vs clean DSL round-trip, broken caret behavior in colored spans, paste handling disasters

4. **Rejected: Textarea overlay** — fatal comment-block line count mismatch (textarea has one line per `!`, display collapses consecutive `!` into a single comment block)

5. **data-idx on `.m4bon-measure` AND `.m4bon-measure-num`** — the div needs it for `_updateActionBar` backward walk, the span needs it for popover trigger. Duplicated but explicit.

## Key Implementation Details

- `data-idx` added in two places in `FormatHTMLRows` (measure div and measure number span) and two in `FormatHTML` (same)
- `focusMeasure(idx)` estimates scroll position via `(charPosition / totalChars) * scrollHeight` — not pixel-perfect but `highlightCursorMeasure` + `scrollIntoView` on the rendered element handles precise visual positioning
- `_updateActionBar` walks backward from the cursor's rendered element to find the nearest `.m4bon-measure` — handles the case where cursor is on a comment block and we need the measure it belongs to
- `clearComments` also removes blank lines immediately preceding the comment block to avoid leaving orphaned empty lines

## Lessons

1. **Measure index threading**: `BuildRows` had access to `mi` (loop index) but wasn't passing it to `MeasureRow`. Adding it was a one-line change but required updating both HTML format paths (`FormatHTMLRows` and `FormatHTML`).

2. **Test string matching**: Tests that checked `'<div class="m4bon-measure">'` broke because that string is now `'<div class="m4bon-measure" data-idx="0">'`. Matching on the prefix `'<div class="m4bon-measure"'` is more robust.

3. **Popover positioning**: `getBoundingClientRect` + viewport clamping works well. The `+4` bottom offset prevents the popover from overlapping the measure number on touch.

4. **Dismiss-on-Escape**: Required storing a `_popoverKeyHandler` reference. Cleaned up in `dismissPopover()`.

5. **Comment detection**: Checking `previousElementSibling.classList.contains('m4bon-comment-line')` is fragile — works for the common case but misses comments separated by blank lines in the rendered output. Good enough for the MVP.
