# Lessons Learned — Session 2026-06-18

## 1. SMF timing: out-of-order NoteOff insertion corrupts delta times

The original `trackBuilder` emitted pending NoteOffs at their scheduled tick using `trackBuilder.add(absTick, msg)`. When `absTick` was earlier than the track's current `lastTick` (because a later event had already been added), the delta was clamped to 0 — placing the NoteOff at the wrong position. This caused notes to sustain past their intended duration, making the tempo sound slow.

Fix: collect all events (NoteOn, NoteOff, metronome) as `{tick, msg}` pairs in a slice, then sort by tick and build the track with proper delta-times from the sorted list.

## 2. Key signature accidentals silently ignored in MIDI pitch resolution

`resolveOctaves` in `pipeline.go` passed `ev.Accidental` directly to `resolvePitch`. For notes without an explicit accidental (like `g` in E major, which should be G♯), `Accidental=0` produced G natural. The key signature was only applied to the render display, not to MIDI pitch assignment.

Fix: add `fifthsToAccidentalMap` and `effectiveAccidental` functions in the parser (mirroring the renderer's logic) and apply key signature accidentals before octave resolution for both `EventNote` and individual chord pitches.

## 3. Scheduler lifecycle: stale callbacks across multiple plays

The first implementation reused `m.transport.Scheduler()` across plays. Old callbacks accumulated, and the scheduler was never stopped. On the second play, `Start()` was a no-op (already running), so new callbacks were registered alongside old ones. The `program.Send()` approach solved the race condition of direct model mutation from goroutines.

Fix: `sched.Stop()` on each fresh play + create a new `macaudio.NewScheduler(m.midiPlayer)` per play. Send messages via `m.program.Send()` instead of mutating model state directly.

## 4. Separate static start position from live playback position

`currentMeasure` was used for both the static start indicator (moved by ↑/↓) and the live playback position (updated by scheduler callbacks). After the first playback, `currentMeasure` pointed to the last measure played. The second play would start from there instead of the saved start position.

Fix: Introduce `startMeasure` (moved by arrows) and `currentMeasure` (updated during playback). Playback always seeks to `MeasureStarts[startMeasure]`. On stop/end, `currentMeasure` resets to `startMeasure`.

## 5. Segment playback enforces the measure range

Using `m.midiPlayer.PlaySegment(start, end)` stops playback automatically at the end of the selected range, and the scheduler's end callback fires at the same `endTime`. Combined, this prevents playback from running past the end indicator even when the scheduler timing is slightly off.

## 6. Rebuild binary after every change

Multiple times the user reported "still broken" when the binary was stale. The old binary didn't include the latest code changes. Adding `go build -o m4bon ./cmd/m4bon/` to the end of every fix cycle prevents this.

## 7. Exported BuildCells for TUI reuse

The TUI needed plain-text measure lines to display alongside the `▶` indicator. `buildCells` was unexported. Exporting it as `BuildCells` allowed the TUI to use `render.BuildCells(measures)` + `render.FormatPlain()`.

## 9. Voice-poly sustain semantics clarified

- `(c - e)` as a first measure is an error — the `-` in voice 2 has no antecedent.
- `(c ; e)` establishes voice 2 with a rest. A subsequent `-` for that voice means "voice exists, silent in this slot" — it should not error.
- `(c d e) | (- - g)` — a traditional chord followed by voice-poly. The individual pitches C and D sustain into the next chord's voice 1 and 2.

The `priorEvents[1]` hard-coding in the legacy sustain path (pipeline.go lines 68, 152) is fragile for these cases. The fix involves: (a) a voice 2-4 fallback if `[1]` is nil, and (b) a `nil` sentinel in `priorEvents` to mark voices that existed as rests.

## 11. Later session (2026-06-19) — Full plan execution

### Lessons from executing `fix-voicepoly-and-medium.md`

**1. `go test ./...` does NOT build the binary.** TUI code has `//go:build darwin && cgo`. Go skips these during `go test`. The CLI binary is never recompiled by tests. Added `Makefile` with `make` (build + test) — always run it after changes.

**2. macOS Scheduler pattern for UI callbacks is fragile.** The `dlsMIDIPlayer.Position()` returns `playStartUs` (non-advancing) once state is `StateIdle`. So scheduler callbacks for remaining measures never fire after the player stops internally. Replaced with elapsed-time polling: `positionMsg` computes current measure via binary search of `MeasureStarts` using `measureAtTime()`.

**3. NumSlots needs GroupIdx guard.** `last.NumSlots++` in sustain-extension paths also fires for cross-group sustains (e.g., `-b` extending a prior group's `a`). Guard with `if last.GroupIdx == gi` so only same-group extensions increment NumSlots.

**4. `normalizePitchInput` was lowercasing directives.** Old code called `strings.ToLower(text)` which turned `BQ.` into `bq.`. After removing ToLower, directive scanning needed case-insensitive checks for both `"b"`/`"B"`, lowercase/uppercase K/M in `stripDirectives`, and uppercase letter handling in `canonicalKey`.

**5. Traditional chord → voice-poly sustain needs virtual events.** `buildPriorEvents` only tracked voice 1 for traditional chords. Added per-pitch virtual `EventNote` entries so voice 2+ sustains in following measures can pick up individual chord pitches.

**6. Golden files need trailing newline matching.** CLI uses `fmt.Println(xml)` (adds newline). Library `Generate()` doesn't. `musicxml/golden_test.go` appends `"\n"` to match.

**7. `*struct{}` with encoding/xml produces `<chord></chord>`, not `<chord/>`.** Semantically identical, kept `<chord></chord>`.

**8. Unkeyed Fraction literals trigger go vet.** `Fraction{1, 2}` → `Fraction{Num: 1, Den: 2}`. Fixed in `standardDurations`.

**9. Constructor wiring needs SlotRest case.** Old code used `EventType(slot.Type)` which handled `SlotNote`, `SlotChord`, and `SlotRest`. Switch to constructors required an explicit `SlotRest` branch.

**10. Use `make` not `go test` for full validation.** Always run `make check` before committing. Test-only validation misses TUI build errors and vet warnings.

## 12. Later session — TUI display, accidental persistence, sustain octave, pickup numbering

### Render `GroupSlots` for cross-group sustain display

Split events (`ev.Split == true`) were skipped entirely in the render, so cross-measure sustains at measure start showed nothing. Mixed groups like `-b` absorbed the sustain into the previous event's duration without creating a display event.

**Fix**: Added `GroupSlots []int` to `MeasureResult` (one entry per beat group recording slot count). The render computes `startSustains = slotCount - nonSplitCount - intraGroupSustains` and prepends that many `-` dashes. The `NumSlots` field on Event already tracks intra-group sustains (same `GroupIdx`), but cross-group sustains leave no event — the slot-count delta fills the gap.

### Accidental persistence within a measure

Standard notation: once `&g` (G♭) appears, all subsequent Gs in the same measure are also flat unless canceled by `%` or a different accidental. The parser had no measure-level state.

**Fix**: `measureLevelAccidental()` in `resolveOctavesMeasures` uses a per-measure `measureAcc map[string]int`. Notes with explicit accidentals or `%` update the map. Notes without check the map before falling through to the key signature.

**Separate `EffAccidental` from `Accidental`**: Added `EffAccidental int` to `Event`. The raw `Accidental` stays for printed accidental symbols (MusicXML `<accidental>`). `EffAccidental` drives `<alter>` and render `noteStyle`. Only explicit accidentals should display a printed symbol — measure-persisted ones should not.

### Cross-measure sustain octave shift

Sustain events copied `OctaveShift` from the prior event. A `^d` (octave-up D♭) sustained across the barline produced a sustain event with `OctaveShift=1`, which `resolvePitch` applied after closest-interval resolution — pushing it an extra octave higher.

**Fix**: Set `OctaveShift: 0` on all three sustain-creation paths (pure-sustain groups, mixed-group cross-measure, voice-poly cross-measure). The sustain continues the same pitch; it must not re-apply the octave shift.

### TUI byte-based truncation

`view.go:99-100` used `line[:m.width-9]` — byte slicing on strings containing ~20-byte ANSI escape sequences. A ~140-byte render line on an 80-column terminal was truncated at byte 71, slicing through `ESC[38;2;15...`, which most terminals interpret as garbled state — swallowing everything after the break (including `;a`).

**Fix**: `visibleLen()` strips ANSI codes, counts visible runes, treats combining marks as zero-width. `truncateVisible()` finds the byte boundary at the Nth visible character without splitting runes or ANSI sequences. Both in `view.go`.

### Pickup measure numbering

`BuildCells` now checks `measures[0].IsPickup` and offsets numbering so pickups are 0 and the first full measure is 1.

## 13. Later session — Leap indicators, subscript toggle, file watching, octave resolution refactor

### Leap indicators from octave marks only

The lilypond "closest interval" rule guarantees that without `^` or `/`, a pitch can never be more than a 4th from the previous. So `OctaveShift > 0` → LeapUp, `OctaveShift < 0` → LeapDown. No MIDI-based interval computation needed. Removed the entire `computeLeaps` / `diatonicStepWithOctave` / `leapFromSteps` machinery — just `leapFromShift(ev.OctaveShift)`.

Combining circumflex (U+0302) for upward leaps, combining macron below (U+0331) for downward. `-ascii-leaps` flag falls back to ANSI overline/underline. Diacritic placed after letter, before subscript.

### `ExplicitNatural` bug in render

`noteStyle` for `EventNote` was hardcoding `false` for `explicitNatural` instead of `ev.ExplicitNatural`. In key of F, `%b` rendered blue (flat) because the key signature's Bb was applied despite the explicit natural. The MIDI was correct — only the color was wrong. Changed to `ev.ExplicitNatural`.

### `diatonicStep` vs `NoteOffsets` confusion

`theory.NoteOffsets` maps letters to *chromatic* offsets (c=0, d=2, e=4, f=5...). Used it as diatonic steps (c=0, d=1, e=2, f=3...), producing wildly wrong leap detection for f (step 5 instead of 3). Need a separate `diatonicSteps` map.

### MIDI-based octave resolution is harmful — removed entirely

`resolvePitch` used `reference / 12` and `candidate = oct*12 + raw` — semitone arithmetic that creates C-centered octave boundaries. Db5 (MIDI 73, octave 6) → Cb5 (MIDI 71, octave 5) appear to be 8 diatonic steps apart (octave 6→5 = -7, plus d→c = -1) even though they're only 2 semitones apart (a 2nd).

**Replacements:**
- `resolveOctave(targetLetter, refLetter, refOctave, octaveShift)` — pure diatonic step distance
- `nextHigherOctave(refLetter, targetLetter, refOctave, octaveShift)` — pure letter comparison
- `midiFromPitch(letter, accidental, octave)` — one-way lookup, never for decisions
- `ResolvedOctave` / `ResolvedOctaves` fields on `Event` — populated during resolution, consumed by MusicXML & render
- `lastOctave` + `lastLetter` maps replace `lastPitch` MIDI map
- `midiToStep` removed from musicxml — uses `ResolvedOctave` + letter/accidental directly

**Key principle:** Pitch identity (letter + accidental + octave) must be resolved without any reliance on MIDI numbers. MIDI is derived from resolved identity, never the reverse.

### `tea.Every` fires once, not repeatedly

BubbleTea's `tea.Every` creates a one-shot timer. For file watching, the callback must always return a non-nil message (even "no change"), and the Update handler must return a new `tea.Every` to restart the cycle. Returning `nil` from the callback swallows the tick and stops the timer.

### TUI file watching and subscripts

- Poll source file `mtime` every 500ms via `tea.Every` + `watchTickMsg` restart loop
- `u` key for manual reload (re-reads file, re-parses, regenerates SMF)
- `o` key toggles octave subscripts (off by default in TUI)
- `showSubscripts` threaded through `render.Render → BuildCells → eventToCells → octaveSubscript`
- `sourceFile` set from `dslLabel` when it's a file path (not "arg" or empty)

## 12. Sustains cause beat overflow: `perNoteDen` must use `posCount`, not `activeCount`

When a beat contains both sustain slots and active notes (e.g., `-d` = 1 sustain + 1 note), `activeCount=1` caused `perNoteDen = totalDen * 1` — giving the active note the entire beat duration. The sustain then extended the prior note by `posNum/posDen`, effectively double-booking the beat. Result: measure overflow (e.g., 4.5 beats in 4/4), barline splits on notes that shouldn't split, and MIDI timing corruption.

**Fix:** `perNoteDen = totalDen * posCount` instead of `totalDen * activeCount`. Each slot (active or sustain) gets exactly `totalDuration / posCount`. The beat no longer overflows.

This is subtle because it only visibly breaks with mixed sustain+active groups. Groups with all-active positions (`ff`, `abc`) were unaffected since `posCount == activeCount`. The bug was hidden until `ff -d` was tested.

## 13. MIDI same-tick NoteOff/NoteOn collision silences repeated notes

When two consecutive notes share the same pitch on the same MIDI channel (e.g., `ff` in `ff -d`), the first note's NoteOff and second note's NoteOn land at the identical absolute tick. The SMF `buildTrackFromEvents` uses stable sort by tick, preserving insertion order (NoteOff before NoteOn), and emits them as `delta=0` events. Some MIDI players ignore or merge same-tick events for identical note/channel pairs, effectively silencing the separation.

**Fix:** In `midi/generate.go`, when a pending NoteOff's scheduled tick equals the current `atTick`, emit it at `offTick - 1` (one tick early). The 1-tick gap (≈0.5ms at 120 BPM) is inaudible but guarantees structural separation.

```go
offTick := pt
if offTick == atTick && offTick > 0 {
    offTick--
}
```

This applies to **all** NoteOff/NoteOn boundaries, not just same-pitch ones — it's a general correctness improvement for any consecutive notes on the same channel.

## 14. Web app editing UX: auto-reformat on valid parse

The web app's textarea was a cramped 60px fixed height with no way to see more than ~2 lines of DSL. Three refinements made editing fluid:

**Resizable + auto-expanding textarea:** Replaced fixed `height: 60px; resize: none` with `min-height: 60px; max-height: 40vh; resize: vertical`. Added `autoResizeTextarea()` that resets to `auto`, reads `scrollHeight`, and clamps to `40vh`. Called on every input, window resize, and state restore. The drag handle sets a baseline; auto-expand handles growth from there.

**Cursor-to-measure highlighting:** `highlightCursorMeasure()` counts `|` before `selectionStart` to find the measure index, applies `.m4bon-cursor` class (subtle blue bg + accent border), and calls `scrollIntoView({ block: 'nearest', behavior: 'smooth' })`. Called on `keyup`, `click`, and after `updateMeasures()` replaces innerHTML. Chose `keyup` over `selectionchange` because `selectionchange` fires too aggressively (every arrow-key navigation).

**Auto-reformat on valid parse:** When the DSL parses without error (debounced, 150ms), `reformatColumns()` rewrites the textarea into canonical one-measure-per-line format with `:H` and `:L` columns vertically aligned. Implemented the parser's `extractDirectivesTail` state machine in JS (state 0=notation, 1=chords after `:H`, 2=lyrics after `:L`). Computes max column widths across all measures, then pads each to align `:H` and `:L` markers. Barlines go at end of line (`|`). Directives on their own line. Only updates the textarea if the reformatted output differs from the current value (guards against reformat loops). Same logic runs on file load.

**Key lesson:** The initial `:H`/`:L` as keyboard-chord-trigger design was a misunderstanding. The user meant `:H` and `:L` as DSL markers (chord symbols and lyrics) that should be visually aligned into columns — exactly what the Go renderer's `BuildRows` already does for display output. Distinguishing between "trigger a command" and "format existing content" is critical in UX design — the user always intended the latter. The JS reformatter mirrors the parser's own token extraction to produce aligned source text, not rendered output.

## 15. Web app recording: synced MIDI playback, count-in, and playback highlight

Recording was rewritten to synchronize mic capture with MIDI playback. Key learnings:

**Count-in must shift `startWall` before MIDI scheduling, not after.** The first version added count-in clicks and shifted `startWall` after the note pre-scan loop had already captured `startWall` at its pre-shift value. This caused measure highlights to be one measure ahead of the actual audio — MIDI notes were scheduled at the old wall-clock reference while the highlight timer used the shifted origin. Fix: move count-in to the very top of `play()`, before any `scheduleNote` calls.

**Count-in must be included in the playback timeout.** The `playbackTimer` (which fires `onPlaybackEnd` → stops recording) was calculated from `(endTick - startTick) * tickToSec` without adding `countInSec`. For full-piece recordings, this cut off the final measure. Fix: `totalSec += countInSec`.

**`measureStarts` from the SMF JSON is already in seconds**, not ticks. The recording playback highlight initially did `measureStarts.map(s => s * tickToSec)`, collapsing all starts to ~0 relative to `audio.currentTime`. All measures appeared to start simultaneously, so the highlight stuck on the last measure. Fix: use `measureStarts` directly as seconds.

**Recording subset highlight needs range offset.** `_recordMeasureSecs` must use the same `measureStarts.map(s => s - rangeOffset)` as `_playMeasureStarts` so the first recorded measure maps to time 0 (+ countInSec) in the audio blob. Without the offset, playback highlight always starts at measure 1 regardless of the recorded range.

**Browser audio processing distorts mic input during playback.** `getUserMedia({ audio: true })` enables echo cancellation, noise suppression, and auto-gain by default. These DSPs are designed for VoIP and distort music when the mic picks up speaker output. Disabling all three (`echoCancellation: false, noiseSuppression: false, autoGainControl: false`) eliminated severe distortion that the TUI never had (CoreAudio gives raw PCM).

**`const startWall += ...` throws "Assignment to constant variable."** Adding count-in required `startWall` to be mutable. Changing `const` to `let` was necessary but easy to miss since `startWall` was declared far from the mutation site.

**MediaRecorder produces usable Blobs for `new Audio()` playback.** No intermediate encoding needed — the WebM/Opus blob from `MediaRecorder` plays directly via `new Audio(URL.createObjectURL(blob))`. The `processRecording()` callback (from `ondataavailable`) stores the blob ephemerally; `playRecording()` plays it with a toggle (first press plays, second stops and rewinds).
