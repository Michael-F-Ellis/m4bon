# Lessons Learned — Session 2026-06-18

## 1. SMF timing: out-of-order NoteOff insertion corrupts delta times

The original `trackBuilder` emitted pending NoteOffs at their scheduled tick using `trackBuilder.add(absTick, msg)`. When `absTick` was earlier than the track's current `lastTick` (because a later event had already been added), the delta was clamped to 0 — placing the NoteOff at the wrong position. This caused notes to sustain past their intended duration, making the tempo sound slow.

Fix: collect all events (NoteOn, NoteOff, metronome) as `{tick, msg}` pairs in a slice, then sort by tick and build the track with proper delta-times from the sorted list.

## 2. Key signature accidentals silently ignored in MIDI pitch resolution

`resolveOctaves` in `pipeline.go` passed `ev.Accidental` directly to `resolvePitch`. For notes without an explicit accidental (like `g` in E major, which should be G♯), `Accidental=0` produced G natural. The key signature was only applied to the render display, not to MIDI pitch assignment.

Fix: add `fifthsToAccidentalMap` and `effectiveAccidental` functions in the parser (mirroring the renderer's logic) and apply key signature accidentals before octave resolution for both `EventNote` and individual chord pitches.

## 3. Scheduler lifecycle: stale callbacks across multiple plays

The first implementation reused `m.transport.Scheduler()` across plays. Old callbacks accumulated, and the scheduler was never stopped. On the second play, `Start()` was a no-op (already running), so new callbacks were registered alongside old ones. The `program.Send()` approach solved the race condition of direct model mutation from goroutines.

Fix: `sched.Stop()` on each fresh play + create a new `macaudio.NewScheduler(m.midiPlayer)` per play. Send messages via `m.program.Send()` instead of mutating model state directly.

## 4. Separate static start position from live playback position

`currentMeasure` was used for both the static start indicator (moved by ↑/↓) and the live playback position (updated by scheduler callbacks). After the first playback, `currentMeasure` pointed to the last measure played. The second play would start from there instead of the saved start position.

Fix: Introduce `startMeasure` (moved by arrows) and `currentMeasure` (updated during playback). Playback always seeks to `MeasureStarts[startMeasure]`. On stop/end, `currentMeasure` resets to `startMeasure`.

## 5. Segment playback enforces the measure range

Using `m.midiPlayer.PlaySegment(start, end)` stops playback automatically at the end of the selected range, and the scheduler's end callback fires at the same `endTime`. Combined, this prevents playback from running past the end indicator even when the scheduler timing is slightly off.

## 6. Rebuild binary after every change

Multiple times the user reported "still broken" when the binary was stale. The old binary didn't include the latest code changes. Adding `go build -o m4bon ./cmd/m4bon/` to the end of every fix cycle prevents this.

## 7. Exported BuildCells for TUI reuse

The TUI needed plain-text measure lines to display alongside the `▶` indicator. `buildCells` was unexported. Exporting it as `BuildCells` allowed the TUI to use `render.BuildCells(measures)` + `render.FormatPlain()`.
