# Lessons Learned — Session 2026-06-17

## 1. Voice-poly chords: `-` and `;` inside `()`

`(c - e)` becomes a 3-voice chord: voice 1 plays C, voice 2 sustains (extends voice 2's prior event), voice 3 plays E. The `-` and `;` inside `()` no longer escape to the main slot array — they accumulate as `ChordEntry` entries with typed slot semantics.

```go
// ChordEntry represents one voice's contribution inside ()
type ChordEntry struct {
    Type        SlotType  // SlotNote | SlotSustain | SlotRest
    Letter, Accidental, OctaveShift
}
```

## 2. Traditional vs voice-poly: automatic detection

`(ceg)` → all entries are notes → traditional chord (`Pitches` field). `(c - e)` → any entry is sustain/rest → voice-poly chord (`Entries` field). The distinction is automatic, no new syntax needed.

## 3. Voice assignment by entry position, not cumulative count

Entry 0 is always voice 1, entry 1 is voice 2, etc. A sustain at entry 1 extends voice 2's prior event — not voice 1's. This means `(c - e)` correctly maps to "C (v1) · sustain (extends v1) · E (v2)" where the C gets extended and E is a new voice.

## 4. Slice pointer staleness bug — use indices, not pointers

`voiceLast map[int]*Event` stored pointers into the `events` slice. When `events = append(events, ev)` caused a reallocation (capacity exceeded), those pointers pointed to the old backing array. Writing through them silently modified freed memory.

Fix: `voiceLastIdx map[int]int` stores slice indices instead. Each access does `last := &events[idx]` to get the current pointer after potential reallocations.

## 5. MusicXML onset ordering — sort by (tick, voice)

Multi-voice events must be emitted in onset order within each measure. The original linear `meas.Events` loop can place a voice-2 note at tick 0 *after* a voice-1 note at tick 960 (the barline split fragment). Verovio interprets the voice-2 note as starting at tick 960 instead of tick 0, so it renders at the wrong position.

Fix: pre-compute per-voice cumulative tick positions, collect `(tick, voice, NoteEl)` tuples, sort by `(tick, voice)` before emitting. Beam detection also uses per-voice tick tracking so voices don't cross-beam.

## 6. TieNext must be voice-aware

The cross-measure tie detection looped backwards through the previous measure's events and set `TieNext` on the *first* note/chord found. With voices, this set a spurious tie on voice 2's G when the cross-measure sustain was for voice 1's C.

Fix: match the `ev.Voice` field when scanning for the event to tie.

## 7. `totalTicks` for overlapping voices

The old `totalTicks` summed all event durations, which is wrong for overlapping voices. A voice-1 half note (960) + voice-2 quarter (480) gave 1440 ticks instead of the actual span of 960.

Fix: track per-voice tick positions and return the maximum tick across all voices.

## 8. Standard Go project layout

Moving from flat `package main` to `cmd/m4bon/main.go` + library root:

```
m4bon/
├── m4bon.go          # package m4bon — public API
├── cmd/m4bon/
│   ├── main.go       # package main — CLI
│   └── *_test.go     # integration tests
├── parser/           # package parser
└── musicxml/         # package musicxml
```

Key consideration: when `go test ./cmd/m4bon/`, the working directory is `cmd/m4bon/`, so relative paths like `./m4bon` and `test/cases/` break. Fix with `TestMain` + `os.Chdir(repoRoot())`. Define `repoRoot()` once in a shared `helper_test.go`.

## 9. Golden file test migration

When moving `golden_test.go` to `cmd/m4bon/`, the glob path `test/cases/*.dsl` becomes relative to `cmd/m4bon/`, not the repo root. `os.Chdir` in `TestMain` fixes this cleanly — all relative paths in the test work from the repo root perspective.

## 10. Verovio MIDI output

`verovio` can output MIDI with `-t midi -o output.mid`. On macOS, `open output.mid` plays via QuickTime Player (non-blocking). The imgcat.sh script saves MusicXML to a temp file, pipes to verovio for SVG rendering, then runs verovio again on the temp file for MIDI playback. Temp files clean up after 5s via background subshell.
